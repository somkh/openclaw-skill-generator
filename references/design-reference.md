## 设计原则速查表

在生成任何 skill 时，对照以下清单自检：

| 原则 | 检查方式 |
|------|---------|
| **职责单一** | 这个 skill 只做一件事吗？ |
| **I/O 有类型** | 每个输入输出都有名称、类型和示例吗？ |
| **错误有结构** | 每个出错场景都返回 `{ status, error }` 格式吗？ |
| **触发词够具体** | description 里有 3-5 个触发短语，以及不触发的情况吗？ |
| **有完整示例** | SKILL.md 里有至少一个完整的输入→输出例子吗？ |
| **无硬编码密钥** | 脚本里没有直接写死 API key 或密码吗？ |
| **引用文件都存在** | SKILL.md 里引用的每个脚本文件都实际创建了吗？ |
| **交互点有强制停止** | 每个需要用户输入的地方都有「强制停止点」块吗？（见下方） |

## 强制停止点写法规范

**适用场景：** 任何需要等待用户输入才能继续的地方。

**为什么需要：** 大语言模型的本能是"把任务做完"。如果只写"等待用户回复"，模型很可能自己替用户回答然后继续执行。强制停止点用明确、重复的禁止语来对抗这种倾向。

**标准写法（直接复制到你的 skill 里）：**

```markdown
> 🛑 **强制停止点**
> 发出上面的[问题/消息/提示]后，**立即停止**。
> 不要[在这里写出最可能发生的越界行为，例如"猜测用户答案"、"进入下一步"、"读取文件"]。
> **等待用户发送下一条消息后再继续。**
```

**填写原则：**
- 第三行的"不要XXX"要写得具体，针对这个停止点最可能出现的错误行为来写
- 每个需要用户输入的地方都要有，不能省略
- 不要把多个停止点合并成一个放在文档末尾——它必须紧跟在触发它的那条输出之后

**正确示例：**

```markdown
向用户发送以下问题：

> 你希望 skill 输出 JSON 还是纯文本？

> 🛑 **强制停止点**
> 发出上面的问题后，**立即停止**。
> 不要替用户选择格式，不要假设用户偏好，不要进入下一步。
> **等待用户发送下一条消息后再继续。**
```

**错误示例（不要这样写）：**

```markdown
向用户发送以下问题，然后等待回复：

> 你希望 skill 输出 JSON 还是纯文本？
```

问题：这种写法对模型约束力不足，模型可能在同一条回复里把问题和"假设用户选了JSON"的后续步骤一起输出。

---

在 Phase 2 中，逐项确认以下 7 条信息已从用户处获得：

| # | 需要知道的 | 如何判断是否已知 |
|---|-----------|----------------|
| ① | 核心功能 | 能用一句话说清楚吗？ |
| ② | 触发时机 | 有 3-5 个具体短语吗？有"不触发"情况吗？ |
| ③ | 输入参数 | 每个参数都有名称、类型、是否必填、示例吗？ |
| ④ | 输出格式 | 有一个具体的成功响应例子吗？ |
| ⑤ | 出错情况 | 至少 3 种错误场景，每种都有处理方式吗？ |
| ⑥ | 外部依赖 | 明确知道需要/不需要哪些外部资源吗？ |
| ⑦ | 具体例子 | 有一个完整的输入→输出演示吗？ |

## 各字段填写规范

### `name` 字段
- 格式：小写字母 + 连字符，不含空格或特殊字符
- ✅ 好例子：`weather-query`、`weekly-report`、`order-status-check`
- ❌ 坏例子：`WeatherQuery`、`weather query`、`查询天气`

### `description` 字段

这是模型决定"是否要调用此 skill"的唯一依据。必须包含：
1. 功能简介（一句话）
2. 触发短语（3-5 个，用中文引号括起来）
3. 触发的上下文条件
4. **不触发**的情况

参见下方「[description 写法对比](#description-写法对比)」。

### 执行步骤（Behavior 部分）

每个步骤必须精确到"可以直接执行"的程度。
参见下方「[执行步骤写法示例](#执行步骤写法示例)」。

## 输入参数类型参考

| 类型关键词 | 含义 | 示例值 |
|-----------|------|--------|
| `string` | 文字/字符串 | `"上海"`、`"2024-01-01"` |
| `number` | 数字 | `42`、`3.14` |
| `boolean` | 是/否 | `true`、`false` |
| `string[]` | 字符串数组 | `["苹果", "香蕉"]` |
| `enum` | 固定选项之一 | `"metric" \| "imperial"` |
| `filepath` | 本地文件路径 | `"/Users/alice/report.csv"` |
| `url` | 网络地址 | `"https://example.com"` |

## 标准错误代码表

| 错误代码 | 使用场景 |
|---------|---------|
| `MISSING_PARAM` | 必填参数没有提供 |
| `INVALID_TYPE` | 参数类型不对（比如应该是数字却传了文字） |
| `OUT_OF_RANGE` | 数字参数超出允许范围 |
| `NOT_FOUND` | 引用的文件、记录或 URL 不存在 |
| `PERMISSION_DENIED` | 没有权限执行此操作 |
| `TIMEOUT` | 外部服务响应超时 |
| `DEPENDENCY_ERROR` | 外部 API 或工具返回了意外错误 |
| `PARSE_ERROR` | 返回的数据格式无法解析 |
| `UNKNOWN` | 未预料到的错误（附上原始错误信息） |

## description 写法对比

### ❌ 坏的写法（模型无法准确触发）

```yaml
description: >
  查询天气信息的 skill。
```

问题：没有触发短语，没有说什么情况用/不用。

---

### ✅ 好的写法

```yaml
description: >
  查询指定城市的实时天气和未来 3 天预报。
  当用户询问天气相关问题时触发，包括："今天天气怎么样"、
  "上海明天会下雨吗"、"帮我查一下北京的天气"、"这周末适合出门吗"、
  "XX 的气温是多少"。
  仅在用户明确询问特定地点的实时或近期天气时使用。
  不要在用户只是在讨论气候、历史天气统计或天气科学知识时触发。
```

## 执行步骤写法示例

### ❌ 坏的写法（太模糊，模型不知道怎么做）

```markdown
## 执行步骤
1. 获取城市名称
2. 调用天气 API
3. 返回结果
```

---

### ✅ 好的写法（每步精确到可执行）

```markdown
## 执行步骤

1. **参数校验**
   - 检查 `city` 参数是否存在且为非空字符串
   - 如果缺失，立即返回：
     ```json
     { "status": "error", "error": { "code": "MISSING_PARAM", "message": "请提供城市名称", "suggestion": "例：上海、北京、纽约" } }
     ```

2. **调用天气 API**
   - 运行脚本：`scripts/fetch_weather.js --city "<city参数的值>"`
   - 脚本会输出 JSON，记录其中的 `temperature`、`condition`、`forecast` 字段

3. **处理 API 返回**
   - 如果脚本 exit code 为 0（成功）：继续 Step 4
   - 如果脚本 exit code 为 1：读取输出中的 `error.code`：
     - `NOT_FOUND`：城市名称无法识别，返回错误提示用户检查拼写
     - `TIMEOUT`：网络请求超时，返回错误建议用户稍后重试
     - 其他：返回 `DEPENDENCY_ERROR`，附上原始错误信息

4. **格式化并返回**
   - 按照「输出格式」中的 success 结构返回数据
```
